### 1. 주택담보대출 심사 백엔드 고도화 설계

#### 🔎 기존 자동화의 한계 정리
- 결과(승인/거절)만 제공
- 판단 근거 불투명
- 감사 대응 어려움
- 모델 편향 설명 부족

단순 자동화가 아니라  
**“설명 가능한 심사 보조 시스템”**이 필요하다는 결론 도출.

---

#### 1️⃣ Explainable 심사 구조 설계

✔ Rule Trace Log 시스템
모든 판단 과정을 기록:

```
[Rule] DSR_Check
입력 DSR: 0.46
내규 한도: 0.40
결과: 초과 → 리스크 플래그
```

→ 결과가 아니라 **판단 과정 자체를 로그화**하는 구조

---

✔ 심사 근거 자동 리포트 (LLM 활용)
- 적용된 규칙 요약
- 초과/미달 항목 설명
- 보완 가능 요소 제안
- 내부 감사용 PDF 자동 생성

→ 단순 스코어링이 아니라 **문서 자동화 시스템**으로 확장 가능

---

✔ AI 판단 + Rule 판단 분리
| 구분 | 결과 |
|------|------|
| 내규 Rule | DSR 초과 |
| AI 모델 | 위험도 0.63 |
| 최종 권고 | 보류 |

→ 사람이 이해할 수 있는 구조로 시각화

---

#### 2️⃣ Config 기반 Rule Engine 설계

#### 문제
- 내규는 자주 변경됨
- 상품/은행별 상이
- 코드 수정은 리스크

#### 해결 방향
Config 기반 구조 설계:

```yaml
loan_type: housing
rules:
  - name: DSR_Check
    threshold: 0.4
    operator: "<="
  - name: LTV_Check
    threshold: 0.7
```
### Rule Engine 기반 심사 구조 설계

#### 🔹 구성 모듈

- **Config Parser**
- **Rule Evaluator**
- **Trace Logger**
- **Decision Aggregator**

→ 단순 자동화 솔루션이 아니라  
**내규를 운영·관리할 수 있는 플랫폼 개념으로 확장 가능**

---

### Human-In-The-Loop 설계

- AI 권고 제공
- 직원 최종 판단

→ 금융 환경에서는  
**AI 대체가 아니라 AI 협업 구조가 현실적**이라는 결론

### 3. AI 강의 정리 – PEFT

#### 🔹 PEFT (Parameter-Efficient Fine-Tuning)

- 전체 모델이 아닌 일부 파라미터만 학습
- 비용 절감
- 메모리 효율성 확보

#### 🔹 Adapter 구조

- Linear layer 삽입
- 계산량 증가
- 선형적 표현 확장 가능
- 비선형 문제 해결에는 한계 존재

→ 금융 특화 LLM 튜닝 시  
**Full Fine-tuning vs PEFT 전략 비교 필요**
