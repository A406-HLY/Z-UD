## 5. SSL 인증서 발급

현재 Nginx 설정(`infra/nginx/conf.d/default.conf`)은 HTTPS(443) 포트 및 TLS 인증서를 필수로 요구합니다. 도커를 실행하기 전에 호스트 머신인 EC2에서 Certbot을 사용해 인증서를 발급해야 합니다. 발급 시 80번 포트가 비어있어야 합니다.

### 5.1. 호스트 내 단독 발급 (초기 1회)

Nginx 실행 전, 호스트 머신의 80번 포트를 활용하여 도메인 인증 및 키 발급

```bash
sudo certbot certonly --standalone -d j14a406.p.ssafy.io
```

### 5.2. Nginx 설정 (`infra/nginx/conf.d/default.conf`)

자동 갱신용 통신 경로 개방과 HTTP 방문자의 HTTPS 리다이렉션을 위해 파일의 최상단에 80번 포트용 `server` 블록을 다음과 같이 추가합니다.

```nginx
server {
    listen 80;
    server_name j14a406.p.ssafy.io;

    # Certbot 자동 갱신용 챌린지 경로
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # 그 외 포트 80 트래픽은 모두 HTTPS(443)로 강제 리다이렉트
    location / {
        return 301 https://$host$request_uri;
    }
}

server {
    listen 443 ssl;
    server_name j14a406.p.ssafy.io;

    # 발급된 인증서 경로
    ssl_certificate /etc/letsencrypt/live/j14a406.p.ssafy.io/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/j14a406.p.ssafy.io/privkey.pem;

    # ...
}
```

### 5.3. Docker Compose 설정 (`infra/compose/docker-compose.yml`)

자동 갱신 모듈 활성화를 위해 메인 컴포즈 파일에 주석 처리되어 있던 Certbot 컨테이너의 주석을 풀고 블록을 완성합니다.

```yaml
  certbot:
    container_name: certbot
    image: certbot/certbot
    entrypoint: "/bin/sh -c 'trap exit TERM; while :; do certbot renew; sleep 12h & wait $${!}; done;'"
    environment:
      TZ: Asia/Seoul
    volumes:
      - /etc/letsencrypt:/etc/letsencrypt
      - /var/www/certbot:/var/www/certbot
    depends_on:
      nginx:
        condition: service_healthy
    restart: unless-stopped
```

인증서 발급이 완료되면 `/etc/letsencrypt` 폴더 하위에 인증서 기반이 생성되며, 이는 Nginx 컨테이너의 볼륨 마운트로 자동 연결됩니다.

## 6. 서비스 구동 (Docker Compose)

작성된 여러 컴포즈 파일을 순차적으로 구동하여 인프라, 백엔드 로직, 프론트엔드 등을 배포합니다. 컴포즈 파일이 있는 `infra/compose` 디렉토리에서 아래 명령을 순서대로 실행합니다.

```bash
# 1. Kafka 클러스터 및 초기 토픽 구성 실행
docker compose -f docker-compose.kafka.yml up -d

# 2. RAG 파이프라인(Vector DB, AI Worker) 실행
docker compose -f docker-compose.rag.yml up -d

# 3. SSO 서버 실행
docker compose -f docker-compose.sso.yml up -d

# 4. 모니터링 시스템(Prometheus, Grafana 등) 실행 - 필요시
docker compose -f docker-compose.monitor.yml up -d

# 5. 메인 시스템(Nginx, Frontend, Backend, Redis, PostgreSQL) 실행
docker compose -f docker-compose.yml up -d
```

## 7. 세부 서비스 세팅

### 7.1. BE

#### 7.1.1. DB 덤프 파일 적용 및 DB 생성

기존 서버에서 추출한 **DDL 파일(`.sql`)**과 **데이터 덤프 파일(`.dump`)**을 새로운 Docker PostgreSQL 환경(`postgres_backend`, `postgres_sso`)에 적용(복원)하는 절차를 안내합니다.

**사전 준비**
1. **파일 확인:** 복원할 대상 서버(호스트)의 현재 경로에 `.sql` 및 `.dump` 파일 4개가 모두 있어야 합니다.
2. **컨테이너 구동:** `postgres_backend`와 `postgres_sso` 컨테이너가 실행 중이어야 합니다.
3. **데이터베이스 생성:** 데이터를 밀어 넣을 빈 데이터베이스가 컨테이너 내부에 생성되어 있어야 합니다.

**1단계: 호스트에서 컨테이너 내부로 파일 복사**
호스트 서버에 있는 백업 파일들을 각 도커 컨테이너 내부의 임시 폴더(`/tmp`)로 복사합니다.

```bash
# Backend 컨테이너로 파일 복사:
docker cp postgres_backend.sql postgres_backend:/tmp/postgres_backend.sql
docker cp postgres_backend.dump postgres_backend:/tmp/postgres_backend.dump

# SSO 컨테이너로 파일 복사:
docker cp postgres_sso.sql postgres_sso:/tmp/postgres_sso.sql
docker cp postgres_sso.dump postgres_sso:/tmp/postgres_sso.dump
```

**2단계: DDL(테이블 구조) 적용하기**
데이터를 넣기 전에 빈 데이터베이스에 테이블, 인덱스 등의 뼈대(스키마)를 먼저 생성합니다. `sso_db` 부분은 실제 SSO용 데이터베이스 이름으로 변경해 주세요.

```bash
# Backend 스키마 생성:
docker exec postgres_backend psql -U root -d zud -f /tmp/postgres_backend.sql

# SSO 스키마 생성:
docker exec postgres_sso psql -U root -d sso_db -f /tmp/postgres_sso.sql
```

**3단계: 덤프(데이터) 복원하기**
뼈대가 만들어졌으므로 실제 데이터를 밀어 넣습니다. 복원 중 에러 발생 시 전체 롤백을 위해 `-1` 옵션을 반드시 포함합니다.

```bash
# Backend 데이터 복원:
docker exec postgres_backend pg_restore -U root -d zud -1 /tmp/postgres_backend.dump

# SSO 데이터 복원:
docker exec postgres_sso pg_restore -U root -d sso_db -1 /tmp/postgres_sso.dump
```

**4단계: 임시 파일 삭제 (Cleanup)**
복원이 성공적으로 완료되었다면, 컨테이너 내부의 저장 공간을 차지하지 않도록 `/tmp` 경로에 복사해 두었던 파일들을 삭제합니다.

```bash
# Backend 임시 파일 삭제:
docker exec postgres_backend rm /tmp/postgres_backend.sql /tmp/postgres_backend.dump

# SSO 임시 파일 삭제:
docker exec postgres_sso rm /tmp/postgres_sso.sql /tmp/postgres_sso.dump
```

### 7.2. Agent

#### 7.2.1. upload-agent 감시 폴더(watch) 생성 및 실행

`upload-agent`는 지정된 감시 폴더를 모니터링하여 신규 PDF 파일을 실시간으로 탐지합니다. 정상적인 파일 감지를 위해 의존성 설치 후 감시 폴더를 생성하고 에이전트를 실행해야 합니다.

**사전 조건**
- Node.js 설치 완료
- npm 사용 가능 환경 구성 완료
- `upload-agent` 프로젝트 소스 확보 완료

**1) 의존성 설치**
`upload-agent` 디렉토리로 이동한 후 아래 명령어를 실행합니다.
```bash
npm install
```

**2) 감시 폴더 생성**
`upload-agent` 루트 디렉토리 하위에 `watch` 폴더를 생성합니다.
```text
upload-agent/
└── watch/
```

**3) 에이전트 실행**
아래 명령어를 실행하여 `upload-agent`를 시작합니다.
```bash
npm run start
```

**4) 동작 확인**
에이전트 실행 후 PDF 파일을 `watch` 폴더에 추가합니다. 파일 추가 시 `upload-agent`가 해당 파일을 실시간으로 감지하는지 확인합니다.

**기대 결과**
- `npm install` 실행 후 필요한 의존성이 정상적으로 설치되어야 합니다.
- `npm run start` 실행 시 에이전트가 정상적으로 구동되어야 합니다.
- `watch` 폴더에 PDF 파일이 추가되면 실시간 감지가 수행되어야 합니다.
- 감지 결과는 콘솔 로그 또는 후속 처리 상태를 통해 확인할 수 있어야 합니다.
