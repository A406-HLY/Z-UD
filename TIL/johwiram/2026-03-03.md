# 📝 2026-03-03 TIL

## 1. 기획 회의 기반 업무 구조 정리

### 1) Flow 차트 설계

- 기획 회의 내용을 기반으로 전체 프로세스를 Flow 차트로 시각화
- 단순 아이디어 정리가 아니라,
  - 입력 데이터 정의
  - 조건 분기 명확화
  - 예외 처리 경로 분리
  - 최종 판단 구조 정리
- “이 기능이 왜 필요한가?”를 흐름 상에서 검증하는 과정 수행

### 배운 점

- 말로 이해한 프로세스는 반드시 흐름도로 검증해야 함
- Flow 차트를 그리는 과정에서 애매했던 부분이 구조적으로 드러남
- 설계는 아이디어 정리가 아니라 **논리 검증 과정**임을 체감

---

## 2. Flow → 문서화 → 기능 명세서 정리

### 1) 기능 정의 문서화

- Flow를 기반으로 단계별 기능 명세 작성
  - 기능 목적
  - 입력값
  - 처리 로직
  - 출력값
  - 예외 케이스
- 단순 설명 문서가 아니라 **개발 가능한 수준의 명세로 정리**

### 2) 데모 구조 정리

- 실제 구현을 가정한 기능 흐름 기반 데모 구조 설계
- 사용자 액션 → 시스템 처리 → 결과 표시 구조 명확화

### 배운 점

- Flow 없이 명세서를 작성하면 추상적이 됨
- 명세서까지 내려가야 기획이 실제 개발 단위가 됨
- 데모 구조를 고려하면서 작성하면 불필요한 기능이 자동으로 걸러짐

## 3. API 구조 리스크 점검 및 대응

### 1) 제공 API 분석

- 제공 예정 API 검토
- 실제 요구사항 대비 기능 범위가 매우 제한적임을 확인
- 데이터 필드, 응답 구조, 확장 가능성 부족 문제 발견

### 2) 추가 조사 및 요청

- 부족한 부분에 대해 추가 자료 조사 진행
- API 확장 요청 및 추가 정보 요청 발송
- 요구 데이터 항목을 명확히 정의하여 요청

### 3) 리스크 대응 전략 수립

- 컨설턴트 미팅을 통해
  - API 제공이 어려울 경우의 대안 시나리오 수립
  - 외부 데이터 대체 방안 검토
  - Mock 데이터 기반 데모 전략 마련

### 배운 점

- 설계 단계에서 API 리스크를 발견한 것은 긍정적
- 외부 의존성이 있는 프로젝트는 반드시 대안 시나리오가 필요
- “기능 설계”보다 중요한 것은 “실행 가능성 검증”
